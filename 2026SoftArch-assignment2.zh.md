# Attribute-Driven Design（ADD）方法

## Step 1：审查输入

ADD 方法的第一步是审查输入，并识别哪些需求将被视为体系结构驱动因素。

## Step 2：通过选择驱动因素建立迭代目标

一个设计轮次通常表现为一系列设计迭代。每次迭代都聚焦于实现一个特定目标。这样的目标通常涉及设计以满足一组驱动因素。

## Step 3：选择一个或多个系统元素进行细化

这一步是核心设计活动开始的地方。你将选择那些与满足特定驱动因素有关的元素。对于绿地开发，可以从建立系统上下文开始，然后选择唯一可用的元素，即系统本身，并通过分解对其进行细化。对于已有系统，或绿地系统后续设计迭代中，通常会选择前面迭代中识别出的元素进行细化。

## Step 4：选择一个或多个满足所选驱动因素的设计概念

这一步要求识别可用于达成本轮迭代目标的设计概念备选方案，并从中选择一个或多个方案。

## Step 5：实例化架构元素、分配职责并定义接口

这一步要求基于所选设计概念实例化架构元素，并为其分配职责。此外，还需要建立元素之间的关系和接口，使它们能够有效协作并交换信息。

## Step 6：绘制视图并记录设计决策

到这一步时，本轮迭代的设计活动已经完成。除了保存视图，也就是你所创建结构的表示之外，还应记录本轮设计迭代中作出的重要决策及其背后的理由，以便后续分析和理解。

## Step 7：分析当前设计，并评审迭代目标和设计目的的达成情况

这一步检查当前的部分设计是否满足本轮迭代目标，并考虑是否还需要额外的设计迭代。

# 使用 ADD 设计成熟领域中的绿地系统：Hotel Pricing System

## 设计目的

该项目可以被视为绿地开发，因为它涉及对现有系统的完全替换。设计活动的目的是作出初始决策，以支持从零开始构建系统。

## 主要功能

| 用例 | 描述 |
| --- | --- |
| HPS-1：登录 | 用户（商业用户或管理员）在登录窗口中提供凭据。系统将这些凭据与用户身份服务进行核对；若验证成功，则允许用户访问系统。登录后，用户只能查询和修改其被授权的酒店。 |
| HPS-2：修改价格 | 用户选择一个其被授权修改价格的特定酒店，并选择希望修改基础价格或固定价格的日期。所有由基础价格计算得到的费率价格会在此时被计算出来。系统允许在实际修改价格前先模拟价格变更。价格被修改后，将被推送到 Channel Management System，并可供外部系统查询。 |
| HPS-3：查询价格 | 用户或外部系统通过用户界面或查询 API 查询给定酒店的价格。 |
| HPS-4：管理酒店 | 管理员添加、变更或修改酒店信息。这包括编辑酒店税率、可用费率和房型。 |
| HPS-5：管理费率 | 管理员添加、变更或修改费率。这包括定义不同费率的计算业务规则。 |
| HPS-6：管理用户 | 管理员修改给定用户的权限。 |

## 质量属性

| ID | 质量属性 | 场景 | 相关用例 | 对客户的重要性 | 实现难度 |
| --- | --- | --- | --- | --- | --- |
| QA-1 | 性能 | 在正常运行期间，某个酒店某个日期的基础价格被修改；该酒店所有费率和房型的价格在少于 100 ms 内发布完成，并可供查询。 | HPS-2 | 高 | 高 |
| QA-2 | 可靠性 | 用户对某个酒店执行多次价格变更；100% 的价格变更都成功发布并可供查询，同时也被 Channel Management System 接收。 | HPS-2 | 高 | 高 |
| QA-3 | 可用性 | 除维护窗口外，价格查询的可用性 SLA 必须达到 99.9%。 | 全部 | 高 | 高 |
| QA-4 | 可扩展性 | 系统初始至少支持每天通过 API 进行 100,000 次价格查询，并应能够扩展到 1,000,000 次，且平均延迟下降不超过 20%。 | HPS-3 | 高 | 高 |
| QA-5 | 安全性 | 用户通过前端登录系统。用户凭据会通过 User Identity Service 验证；登录后，用户只能看到其被授权使用的功能。 | 全部 | 高 | 中 |
| QA-6 | 可修改性 | 系统新增一种不同于 REST 的价格查询端点协议，例如 gRPC。该新端点不需要修改系统核心组件。 | 全部 | 中 | 中 |
| QA-7 | 可部署性 | 应用在开发过程中从一个非生产环境迁移到另一个非生产环境时，不需要修改代码。 | 全部 | 中 | 中 |
| QA-8 | 可监控性 | 系统运维人员希望在运行期间度量价格发布的性能和可靠性。系统提供一种机制，使得这些度量可以按需 100% 收集。 | HPS-2 | 中 | 中 |
| QA-9 | 可测试性 | 系统及其元素应 100% 支持在不依赖外部系统的情况下独立进行集成测试。 | 全部 | 中 | 中 |
